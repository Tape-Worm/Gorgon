Remarks

This stream type is similar to the MemoryStream object, except that MemoryStream uses an array of bytes as a backing store. This object uses native unmanaged memory to hold its data, and accesses it through the use of either unsafe pointers or IntPtr for those .NET languages that don't support raw pointers (e.g. Visual Basic.NET).

The object supports a multitude of reading/writing options for primitive data, array data, and even generic types to the stream. For generic types being read or written, the type must be decorated with a StructLayoutAttribute and a Sequential or Explicit value. Otherwise, .NET may rearrange the members of the type and the data being read/written will not serialize as expected. Also, for the Read/Write methods, the generic type must not use any marshalling (MarshalAsAttribute) or reference types. Only primitive types and value types (structs) are allowed.

If data needs to be marshalled, then the object provides three members: ReadMarshalT, WriteMarshalT(T, Boolean) and MarshalArrayToStreamT(T) for marshalling reference types to and from the stream. The rules about having the StructLayoutAttribute still apply, and additionally, a size, in bytes of the value should be supplied to the attribute along with the LayoutKind. Because these methods may cause performance to suffer, it is recommended that data be written directly on a per-member basis instead of using these functions if performance is absolutely critical.

This object will also provide native access to an Array by pinning the array and returning a pointer to the handle on the first element via the GCHandle type. Because of how .NET memory management reorders and compacts memory, it is recommended that any stream access to an Array be limited to very short durations or else memory fragmentation could become an issue.

Existing pointers to native memory can be used when constructing this stream type via the GorgonDataStream(IntPtr, Int32) (or using the constructor with the raw void * pointer). The data stream will read and write from this pointer, but it will not take ownership of it, and thus will not free the memory associated with it when it is done. That responsibility is up to the caller of the data stream.

 Important

When using this stream to allocate a new block of memory, it is very important that Dispose(Boolean) method is called when finished using this object. Failure to do so can cause memory leaks due to the inability of the garbage collector to handle the native memory allocated by this stream. The exception to this rule is when this stream is applied to an existing pointer. In those cases, calling Dispose is not necessary as no memory has been allocated by the stream, and the pointer is not owned by the stream.

 Caution

Like most stream based types, it is not safe to use the same instance of this type from multiple threads. Doing so may cause performance degradation, or worse, memory corruption. If multi threading is a possibility and native memory access is required, then the GorgonPointerBase object provides better functionality for working with native memory directly while providing functions that are better suited to thread safety.

A better interface to dealing with unmanaged memory is through the GorgonPointerBase class.

 Caution

Gorgon will be phasing out it's internal usage of this type in favour of more direct access to memory via the GorgonPointerBase
